Actúa como Arquitecto de Software Senior y Desarrollador Full Stack.

Necesito ajustar mi Sistema de Gestión de Soporte Técnico para que tenga un seguimiento completo de tickets en todo el sistema, desde que el usuario final crea el ticket hasta su conclusión.

Stack del proyecto:
- Frontend: React + TypeScript + Tailwind CSS
- Backend: Node.js + Express + TypeScript
- Base de datos: PostgreSQL + Sequelize

Antes de modificar, analiza la estructura actual del proyecto y reutiliza lo que ya existe.

OBJETIVO PRINCIPAL:
Implementar un flujo completo y controlado de tickets con trazabilidad, roles, seguimiento, asignación, prioridades y reportes PDF con un único modelo visual.

==================================================
FLUJO GENERAL DE TICKETS
==================================================

El flujo debe ser así:

1. Usuario final crea un ticket.
2. El ticket queda en estado "Abierto" o "Pendiente de asignación".
3. El administrador revisa los tickets generados.
4. Solo el administrador puede:
   - Asignar el ticket a un técnico.
   - Definir la prioridad del ticket.
   - Reasignar el ticket si es necesario.
5. El técnico solo puede trabajar los tickets asignados a él.
6. El técnico debe seguir el proceso hasta la conclusión:
   - Revisar ticket.
   - Cambiar estado a "En proceso".
   - Registrar observaciones.
   - Registrar acciones realizadas.
   - Registrar solución aplicada.
   - Marcar como "Resuelto".
7. El administrador puede revisar y cerrar el ticket.
8. El sistema debe guardar todo el historial del ticket.

==================================================
ROLES Y PERMISOS
==================================================

Usuario final:
- Crear tickets.
- Ver solo sus tickets.
- Ver el estado de sus solicitudes.
- No puede elegir prioridad.
- No puede asignar técnico.

Administrador:
- Ver todos los tickets.
- Asignar técnico.
- Cambiar prioridad.
- Cambiar estado.
- Reasignar técnico.
- Cerrar tickets.
- Ver historial completo.
- Generar reportes.

Técnico:
- Ver solo tickets asignados a él.
- Cambiar estado operativo.
- Agregar seguimiento.
- Registrar diagnóstico.
- Registrar solución.
- Marcar ticket como resuelto.
- Generar reportes operativos si tiene permiso.

PRIORIDAD DEL TICKET
==================================================

Solo el administrador puede definir o modificar la prioridad.

Prioridades:

- Media

Cuando el usuario final crea un ticket, la prioridad debe quedar vacía, sin asignar o con valor "Por definir".

- Prioridad asignada, si aplica
- Comentario u observación
- Solución aplicada, si aplica

Ejemplos de acciones:

- Ticket creado
- Ticket asignado

- Estado actualizado
- Comentario agregado
- Diagnóstico registrado

==================================================

Debe existir un solo modelo visual de reporte PDF para todo el sistema.

No crear diseños diferentes para cada módulo.

El contenido puede cambiar según el reporte, pero el formato visual debe ser único.

Todos los reportes deben incluir:

- Logo institucional

- Nombre del sistema
- Título del reporte

- Datos, tablas, indicadores o resumen según el tipo de reporte.

- Rol del usuario.
- Fecha y hora de generación.
- Cantidad de registros.
- Número de página.
- Espacio para firma si corresponde.
